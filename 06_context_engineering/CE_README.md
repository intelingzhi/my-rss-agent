
# LLM 应用开发：上下文工程核心笔记

## 1. 为什么需要上下文工程？

上下文工程的核心是解决 LLM 的“记忆问题”。通过对比有无上下文工程的场景，可以清晰地看到其必要性。

| 场景 | 没有上下文工程 | 有上下文工程 |
| :--- | :--- | :--- |
| “接着上面继续写” | 不知道上面是什么 | **记得**上一段代码 |
| “刚才那个 Bug 呢” | 忘了 | **记得**之前的讨论 |
| “我们之前说的那个方案” | 不知道哪个方案 | **记得**完整讨论 |
多轮对话 | 只有最后一轮 | 记住完整对话历史



## 2. 核心概念辨析

### 2.1 本质区别：RAG vs. 上下文工程

这是两个经常被混淆，但解决问题的层面完全不同的概念。

-   **RAG (检索增强生成)**：解决 **“知识问题”**——**Agent 知道什么（静态知识）**。例如，让 AI 基于公司内部文档回答问题。
-   **上下文工程**：解决 **“记忆问题”**——**Agent 记得什么（动态对话）**。例如，让 AI 记得用户五分钟前提到过的名字。

### 2.2 上下文工程的三个核心概念

| 概念 | 作用 | 实现方式 |
| :--- | :--- | :--- |
| **消息 (Messages)** | 存储原始的对话历史 | SQLite 数据库、Redis 等 |
| **记忆 (Memory)** | 从消息中提取关键信息，形成长期记忆 | 摘要/关键点提取 (LLM) |
| **压缩 (Compression)** | 缩短上下文长度，减少 Token 消耗 | LLM 摘要/截断/卸载 |

## 3. 为什么需要压缩？

LLM 存在**上下文长度限制**，并且 Token 消耗直接与**费用**挂钩。

### 3.1 问题示例
一个 100 轮的对话，如果不加处理，Token 消耗会迅速累积。

```python
# 一个 100 轮对话的消息列表
messages = [
    {"role": "user", "content": "帮我写个 Hello World"},  # ~10 tokens
    {"role": "assistant", "content": "print('Hello World')"},  # ~8 tokens
    # ... 100 轮后 = 10000+ tokens
]

# 费用计算（假设 1000 tokens = $0.01）
# 10000 tokens = $0.1/次请求
# 如果每天 100 次对话 = $10/天
```

这就是为什么必须对历史对话进行**压缩**。

## 4. 压缩方案深度解析

### 4.1 压缩的三种方式

| 方式 | 类型 | 说明 | 优点 | 缺点 |
| :--- | :--- | :--- | :--- | :--- |
| **截断 (Truncation)** | 有损 | 直接丢弃最早或最不相关的消息 | 实现简单，速度快 | 可能丢失重要信息 |
| **摘要 (Summarization)** | 有损 | 用 LLM 将多轮对话压缩成一段摘要 | **保留核心信息**，压缩率高 | 可能丢失细节，需额外调用 LLM |
| **卸载 (Offloading)** | 无损 | 将历史消息存储到外部数据库，需要时再加载 | 不丢失任何信息 | 需要时重新加载速度慢，实现复杂 |

- **本课重点**：使用 **LLM 做摘要压缩**，在保留核心信息和压缩率之间取得最佳平衡。

### 4.2 压缩算法详解
核心思路：
- 找到用户最后一次发言的位置
- 从该位置到最新消息，全部压缩成摘要
- 用压缩后的摘要替换原始消息

### 4.3 压缩触发条件


| 阈值 (Token限制) | 适用场景 | 核心特点与说明 |
| :--- | :--- | :--- |
| **4000 tokens** | **长对话、多轮讨论** | 适合需要维持长时间对话记忆的场景，能够处理包含多轮交互的历史记录。 |
| **8000 tokens** | **标准对话** | 大多数通用任务和日常对话的平衡点，足以处理中等长度的文档和复杂指令。 |
| **16000 tokens** | **代码生成** | 需要更多上下文来理解整个代码库的结构、函数调用关系和前文逻辑，避免生成中断或重复。 |

> **注意**：随着模型迭代（如 GPT-4 Turbo、Claude 3 等），上下文窗口正在变得越来越大（如 128K、200K）。上述阈值划分主要基于经典模型的通用建议，在实际应用中可根据具体任务灵活调整。


### 4.4 什么时候压缩
```python
# 策略 1：每次请求前检查
def before_request(messages):
    if should_compress(messages):
        messages = compress_messages(messages, llm)
    return messages

# 策略 2：每次响应后检查
def after_response(messages, response):
    new_messages = messages + [response]
    if should_compress(new_messages):
        new_messages = compress_messages(new_messages, llm)
    return new_messages
```
